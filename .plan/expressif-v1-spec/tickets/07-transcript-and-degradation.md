# Pin transcript representation and text degradation

## Question

Every widget needs a canonical text form. Pin what the *model* sees in later turns after a widget rendered and the user interacted; what survives context compaction; what the transcript file records; and how the same agent session degrades when no expressif surface is attached (plain terminal fallback — the skill/plugin must not break the CLI experience). Decide whether the text form is generated by the plugin, the core, or specified per primitive.

## Answer

Grilled 2026-07-12, within the channel rules MCP Apps fixes (model sees tool result `content`, `ui/message`, `ui/update-model-context`; never `structuredContent` or raw interactions — [MCP Apps validation](../assets/mcp-apps-validation.md), finding 5) and the harness-owned persistence decided in [Define the harness adapter contract](04-define-adapter-contract.md).

### Generator: specified per primitive, core enforces

- **Markdown+** — the canonical text *is* the markdown (identity).
- **Structured inputs** — core-generated deterministically from schema + answer; uniform across all plugins for free.
- **Sandboxed HTML** — agent-supplied: the render tool requires a text summary argument (alt-text discipline, schema-enforced — a render call without one is invalid).
- Plugin tools beyond the primitives fall under the blanket rule (canonical text in result `content`), checked at first-party plugin review alongside ticket 05's budgets.

### Model-facing representation: the record-of-consequence convention

The tool result carries the **outcome, not the UI**: terse, self-contained plain text — what was asked, what was chosen ("Asked: deploy target? (staging / prod / cancel) — user chose: prod"). No UI narration, no JSON dumps. **Self-containment is the hard rule**: the text must carry every decision-relevant fact without the widget existing. Compaction survival then needs no mechanism of its own — both harnesses compact from message/result text, so whatever the convention puts there is what survives. Post-result state changes flow `ui/update-model-context` → adapter `contextInjection` as one-line deltas, **coalesced and injected lazily before the next agent turn** (never mid-turn); only state the plugin marks model-relevant is forwarded.

### Transcript and rehydration: frozen replay, no sidecar

For all three primitives the harness transcript alone determines a faithful frozen replay: the tool call *input* holds the widget definition (HTML source, question schema), the tool *result* holds the outcome, deltas live in injected context messages. So **v1 keeps no widget-state sidecar** — reopening a session re-renders each completed widget read-only from tool input + result; ephemeral in-widget state (scroll, unsubmitted drafts) is lost by design, matching MCP Apps, which deferred state persistence and offers no channel for it. This closes the caveat [Define the harness adapter contract](04-define-adapter-contract.md) deferred here: widget-state replay is *not needed*, the convention makes the transcript sufficient. Still-pending widgets on reopen belong to [Pin return-channel semantics](06-return-channel-semantics.md). If a future plugin needs durable widget state, that is a loud contract/plugin revision when MCP Apps state persistence lands.

### Degradation: canonical trace on detach, absence does the rest

A plain-terminal session is already safe by prior decisions: [Decide the agent-awareness mechanism](05-decide-agent-awareness.md) removes preamble, proxy tools, and pack skills (the agent cannot attempt widgets, so plugins cannot break the CLI); history reads as self-contained outcome text; native questions degrade natively (the harness's own CLI/TUI renders them). The one new rule: **no widget vanishes from the record unexplained**. On detach with pending widgets, the rejection carries a model-visible explanation — via the deny/reject message where the `denyMessage` capability holds, else a context injection immediately before the reject (opencode's bare question-reject) — stating the surface detached unanswered and the agent should re-ask in plain text if still needed. Inject-then-reject-then-close ordering is adapter mechanics.
